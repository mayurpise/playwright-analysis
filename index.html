Playwright is Microsoft's modern end-to-end testing framework released in 2020. Built by former Puppeteer developers, it provides reliable browser automation across Chromium, Firefox, and WebKit with a single unified API.

üåê Cross-Browser

Chromium, Firefox, WebKit ‚Äî same API, consistent behavior

üñ•Ô∏è Cross-Platform

Windows, macOS, Linux ‚Äî local or CI, headless or headed

üíª Multi-Language

TypeScript, JavaScript, Python, .NET, Java bindings

Auto-Wait Mechanism

Automatically waits for elements to be actionable before performing actions, eliminating flaky tests caused by timing issues.

Web-First Assertions

Assertions auto-retry until conditions are met, designed specifically for dynamic web content.

Browser Context Isolation

Each test runs in isolated browser context (like fresh profile) ‚Äî full isolation, zero overhead, milliseconds to create.

Powerful Tracing

Capture execution traces, DOM snapshots, screenshots, videos for debugging. Time-travel through test execution.

Unlike Selenium's WebDriver protocol, Playwright communicates directly with browser engines via Chrome DevTools Protocol (CDP) for Chromium and similar low-level protocols for Firefox and WebKit.

Locators are the central piece of Playwright's auto-waiting and retry-ability. They represent how to find elements at any moment, not a snapshot reference.

Recommended Locator Priority:

  1. 1
    getByRole() ‚Äî Accessibility attributes (best practice)
  2. 2
    getByLabel() ‚Äî Form field labels
  3. 3
    getByPlaceholder() ‚Äî Input placeholders
  4. 4
    getByText() ‚Äî Visible text content
  5. 5
    getByTestId() ‚Äî data-testid attributes
// User-facing locators (recommended)
await page.getByRole('button', { name: 'Submit' }).click();
await page.getByLabel('Email').fill('test@example.com');

// Chaining and filtering
const product = page.getByRole('listitem')
  .filter({ hasText: 'Product 2' });
await product.getByRole('button', { name: 'Add to cart' }).click();

Before every action, Playwright performs actionability checks automatically:

‚úì Attached to DOM
‚úì Visible
‚úì Stable (not animating)
‚úì Receives events
‚úì Enabled
‚úì Editable (for fill) 
// No manual waits needed - Playwright handles timing
await page.getByRole('button', { name: 'Submit' }).click();

// Web-first assertions auto-retry
await expect(page.getByText('Success!')).toBeVisible();

// Built-in timeout with configurable default
await locator.click({ timeout: 10000 }); // 10 seconds

  • ‚Ä¢ DOM Snapshots ‚Äî Before/after state for every action

  • ‚Ä¢ Console Logs ‚Äî Browser and test console output

Released March 2025 ‚Äî Playwright MCP is a server that enables LLMs (Claude, Copilot, etc.) to control real browsers using the accessibility tree instead of screenshots or vision models.

üéØ Key Innovation

Uses accessibility snapshots ‚Äî structured, text-based webpage representation. No vision models needed, fast & deterministic.

üîß Integration

Works with Claude Desktop, Cursor IDE, VS Code Copilot, Windsurf, and any MCP-compatible client.

Quick setup for various AI tools:

// Standard MCP Configuration
{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": ["@playwright/mcp@latest"]
    }
  }
}

// With headless mode for CI
{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": ["@playwright/mcp@latest", "--headless"]
    }
  }
}

// Run standalone server
npx @playwright/mcp@latest --port 8931

Choose Playwright When:

  • ‚Ä¢ Building modern web apps (SPAs, React, Vue, Angular)
  • ‚Ä¢ Need reliable cross-browser testing with minimal flakiness
  • ‚Ä¢ Want AI-powered test generation (MCP integration)
  • ‚Ä¢ Multi-tab, multi-context, or multi-user scenarios

Choose Selenium When:

  • ‚Ä¢ Legacy browser support required (IE11, older versions)
  • ‚Ä¢ Large enterprise with existing Selenium infrastructure
  • ‚Ä¢ Team uses Ruby, PHP, or other non-Playwright languages
  • ‚Ä¢ Need Appium integration for native mobile apps

Choose Cypress When:

  • ‚Ä¢ Frontend-focused team (JS/TS only)
  • ‚Ä¢ Want fastest possible feedback loop during development
  • ‚Ä¢ Single-tab testing is sufficient
